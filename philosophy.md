# AI-DLC Philosophy

> AI-Driven Development Lifecycle の思想と原則

## コアコンセプト

**「AIが駆動し、人間が舵を取る」**

AI-Driven Development Lifecycle は、AIを開発の実行エンジンとして中心に据え、人間は意思決定と品質保証に集中するワークフローである。

「AIを使う開発」が既存の人間中心プロセスにAIを差し込むのに対し、AI-DLC は**プロセス自体をAI前提で再設計する**。

### 「AIを使う開発」との違い

**AIを使う開発（従来）:**

```
人間: 「このバグ直して」 → AI: コード修正 → 人間: マージ
```

人間のワークフローにAIを差し込んだだけ。

**AI-Driven な開発（AI-DLC）:**

```
人間: 「こういう問題を解決したい」
  → AI: 問題を分析し、複数のアプローチを提案
  → 人間: 方向性を選択
  → AI: 要件を具体化し、設計を生成し、実装し、テストまで一気通貫で実行
  → 人間: 最終確認・承認
```

### この違いが生む根本的な変化

1. **単位の変化**: 「この関数を書いて」ではなく「この問題を解決して」
2. **粒度の変化**: 人間はマイクロマネジメントしない。AIに大きな塊を渡す
3. **フィードバックの変化**: コードレベルではなく「動くもの」レベルで判断する

## 3原則

| 原則             | 意味                                                                       |
| ---------------- | -------------------------------------------------------------------------- |
| **AI-Driven**    | AIが具体化・設計・実装・テストを主体的に駆動する。人間は最終判断に集中する |
| **Reproducible** | 標準化されたプロセスにより、誰がやっても同品質のアウトプットが出る         |
| **Adaptive**     | 技術スタック・規模・ドメインを問わず適用可能。フェーズの取捨選択も自由     |

## 人間とAIの役割分担

| 責務             | 人間               | AI                     |
| ---------------- | ------------------ | ---------------------- |
| ビジョン・ゴール | 決める             | -                      |
| 要件の具体化     | 承認する           | 提案・生成する         |
| 設計             | 方向性を判断する   | 選択肢を生成・推奨する |
| 実装             | レビュー・承認する | コードを書く           |
| テスト           | 受け入れ判断する   | テストを設計・実行する |
| デプロイ         | Go/No-Go判断する   | 自動化する             |

**キーワード**: 人間は「承認ゲート」、AIは「実行エンジン」

## 粒度の可変性

AIに渡す作業単位はフェーズで変わる:

```
構想フェーズ  → プロダクト単位（大きく考える）
設計フェーズ  → ユースケース単位（構造を決める）
実装フェーズ  → 機能/タスク単位（確実に作る）
検証フェーズ  → チェック項目単位（漏れなく確認する）
```

## 承認ゲートの設計哲学

- **初期は多め、成熟したら減らす** (Trust but Verify → Trust and Spot-check)
- 各フェーズの切り替え時に必ず人間の承認を入れる
- フェーズ内でも重要な意思決定ポイントでは承認を求める
- **ゲートは「ブレーキ」ではなく「ハンドル」** — 方向を微調整する場

## 失敗に対する考え方

- AIは必ず間違える。重要なのは**間違いの検知が早いこと**
- 各フェーズの成果物を明確にし、成果物のレビューで品質を担保する
- やり直しのコストを最小化する（小さく作って早く確認する）
